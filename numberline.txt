================================================================================
1. WHAT IS THE NUMBERLINE GAME?
================================================================================

NumberLine is the SIMPLEST of the four gym_cards environments in the RL4VLM
project. It serves as the basic test case / debugging environment for the
entire VLM + RL pipeline.

THE GAME:
  - You have a number line from 0 to max_position (default = 5, so 0-5)
  - At the start of each episode, a RANDOM "Current" number and a RANDOM
    "Target" number are placed on the line (they are always different)
  - The agent sees an IMAGE (300x300 RGB) showing:
      "Target: X"
      "Current: Y"
    (white background, black text, DejaVuSans-Bold font, size 36)
  - The agent must choose ONE of TWO actions each step:
      Action 0: "-" (subtract 1 from current position)
      Action 1: "+" (add 1 to current position)
  - Goal: Move "Current" to match "Target"
  - The episode ends when Current == Target (success) or after 2*max_position
    steps (truncation/failure)

WHY THIS TASK MATTERS:
  - Tests VISUAL recognition (can the VLM read numbers from an image?)
  - Tests ARITHMETIC reasoning (should I go up or down?)
  - Tests MULTI-STEP decision making (needs multiple correct actions in a row)
  - Deterministic (no randomness in transitions), so performance directly
    measures the agent's reasoning quality
  - Simple 2-action space, so it isolates vision+reasoning from complex
    action selection

PAPER RESULTS FOR NUMBERLINE:
  - GPT-4V:     65.5% success rate
  - Gemini:     51.7% success rate
  - CNN+RL:     87.1% success rate
  - LLaVA-SFT:  50.2% success rate (SFT alone, no RL)
  - RL4VLM:     89.4% success rate (SFT + RL, BEST result)
  
  CoT ablation:
  - With CoT:    89.4%
  - Without CoT: 26.9%
  - Difference:  +62.5% (CoT is CRITICAL for NumberLine)

2. HOW THE ENVIRONMENT WORKS (CODE-LEVEL)
================================================================================

FILE: gym-cards/gym_cards/envs/numberline.py

CLASS: NumberLineEnv(gym.Env)

INITIALIZATION (__init__):
  - max_position = 5 (number line goes from 0 to 5)
  - action_space = Discrete(2)  -> {0: move left/"-", 1: move right/"+"}
  - observation_space = Box(0, 255, shape=(300,300,3), dtype=uint8)
  - Randomly picks start_position and goal_position from [0, max_position]
  - If start == goal, shifts goal by 1 (mod max_position)
  - position = start_position
  - steps_made = 0

RESET (reset):
  - Re-randomizes start_position and goal_position
  - Ensures they are different
  - Resets position = start_position, steps_made = 0
  - Returns: (observation_image, info_dict)
  - info = {"Target": goal_position, "Current": position}

STEP (step):
  - Increments steps_made
  - Calculates prev_distance = |position - goal|
  - If action==0 and position > 0: position -= 1
  - If action==1 and position < max_position: position += 1
  - Calculates current_distance = |position - goal|
  - REWARD:
      If current_distance == 0:     reward = +1, done = True (SUCCESS!)
      If current_distance >= prev:   reward = -1 (moved wrong way or stayed)
      Else:                          reward = 0 (moved closer but not there yet)
  - TRUNCATION: If steps_made >= 2 * max_position and not done -> truncate
  - Returns: (observation, reward, done, truncate, info)

OBSERVATION (_get_observation):
  - Creates a 300x300 white PIL Image
  - Draws "Target: X" at position (30, 60) in black, font size 36
  - Draws "Current: Y" at position (30, 180) in black, font size 36
  - Uses font: 'dejavu/DejaVuSans-Bold.ttf'
  - Converts to numpy array and returns

EXAMPLE EPISODE (max_position=5):
  Reset -> Target=4, Current=1, Image shows "Target: 4" and "Current: 1"
  Step(1/"+") -> Current=2, reward=0, done=False  (moved closer)
  Step(1/"+") -> Current=3, reward=0, done=False  (moved closer)
  Step(1/"+") -> Current=4, reward=+1, done=True  (REACHED TARGET!)
  Total reward = +1 (success)

REGISTRATION (gym-cards/gym_cards/__init__.py):
  register(
      id='gym_cards/NumberLine-v0',
      entry_point='gym_cards.envs:NumberLineEnv',
      max_episode_steps=300,
  )

  So you use it as: env = gym.make('gym_cards/NumberLine-v0')

4. PHASE 1: SUPERVISED FINE-TUNING (SFT)

4A. WHERE IS THE SFT DATA? (HUGGING FACE)
------------------------------------------

The SFT data is hosted at:
  https://huggingface.co/LEVI-Project/sft-data/tree/main

This Hugging Face repository contains:
  - JSON files with instruction-following data for each task
  - Image folders with the corresponding observation images

For NumberLine specifically, the data contains:
  - A JSON file with entries like:
    {
      "id": "unique_id",
      "image": "path/to/image.png",
      "conversations": [
        {"from": "human", "value": "<image>\n<the prompt text>"},
        {"from": "gpt", "value": "<the expected JSON response>"}
      ]
    }
  - An image folder with all the 300x300 PNG observation images

HOW TO DOWNLOAD THE SFT DATA:

  Option 1 - Web browser:
    Go to https://huggingface.co/LEVI-Project/sft-data/tree/main
    Download the JSON files and image folders manually

  Option 2 - Git LFS:
    git lfs install
    git clone https://huggingface.co/LEVI-Project/sft-data

  Option 3 - Hugging Face CLI:
    pip install huggingface-hub
    huggingface-cli download LEVI-Project/sft-data --local-dir ./sft-data

The data was generated by an EXPERT SOLVER (not by humans or GPT-4V):
  - For each task, the solver plays the game optimally
  - At each step, it records the observation image and constructs the
    "correct" JSON response including CoT reasoning and action
  - This produces 20k-50k samples per task for gym_cards
  - Two versions exist: WITH CoT and WITHOUT CoT

4B. SFT DATA FORMAT
--------------------

The SFT JSON file follows the LLaVA fine-tuning format. Each entry looks like:

  {
    "id": "numberline_00001",
    "image": "numberline/episode_1_step_0.png",
    "conversations": [
      {
        "from": "human",
        "value": "<image>\nYou are playing a game called number line. You will
                  see a target number and a current number in the image. And your
                  goal is to move the current number closer to the target by
                  choosing either adding or subtracting one to the current number.
                  Your response should be a valid json file in the following format:
                  \n{\n \"current number\": \"x\", \n\"target number\": \"x\",
                  \n\"thoughts\": \"{first read out the current and target number,
                  then think carefully about which action to choose}\",
                  \n\"action\": \"-\" or \"+\" \n}"
      },
      {
        "from": "gpt",
        "value": "{\n\"current number\": \"1\",\n\"target number\": \"4\",
                  \n\"thoughts\": \"The current number is 1 and the target is 4.
                  Since 4 > 1, I need to add to move closer.\",\n\"action\": \"+\"\n}"
      }
    ]
  }

KEY OBSERVATIONS ABOUT THE DATA:
  - The "human" message contains the <image> token + the prompt text
  - The "gpt" message is the expected model output (CoT + action in JSON)
  - The image path is relative to the --image_folder argument
  - The CoT version includes "current number", "target number", "thoughts"
    fields BEFORE the "action" field
  - The action-only version would just have the "action" field

4D. WHAT THE SFT SCRIPT DOES INTERNALLY
----------------------------------------

1. LOADS BASE MODEL:
   - Downloads liuhaotian/llava-v1.6-mistral-7b from HuggingFace Hub
   - This is a ~14GB model with:
     * Mistral-7B LLM backbone
     * CLIP ViT-L/14-336 vision encoder
     * MLP projector (maps 1024-dim CLIP features to 4096-dim Mistral space)

2. LOADS TRAINING DATA:
   - Reads the JSON file
   - For each entry, loads the image from image_folder
   - Tokenizes the conversations using the LLaVA tokenizer
   - The <image> token gets replaced with 576 visual tokens (24x24 patches
     from CLIP's output for a 336x336 image)

3. TRAINS FOR 1 EPOCH:
   - Uses DeepSpeed ZeRO-3 (shards model across GPUs)
   - Batch size: 16 per device
   - Learning rate: 2e-5 with cosine decay and 3% warmup
   - Mixed precision: BF16
   - Gradient checkpointing enabled (saves memory)
   - Loss: standard next-token prediction cross-entropy loss on the
     "gpt" response tokens only (human/system tokens are masked)

4. SAVES CHECKPOINT:
   - Saves complete model weights to --output_dir
   - Includes: model weights, tokenizer, config, mm_projector

5. WHAT'S BEING TRAINED:
   - ALL parameters of the model are fine-tuned (full SFT, not LoRA)
   - This includes the vision encoder, projector, and LLM backbone
   - Total trainable parameters: ~7.5 billion

4E. SFT OUTPUT / CHECKPOINT
----------------------------

After SFT completes, you get a DIRECTORY (not a single file) at --output_dir:

  /your/sft/checkpoint/
    ├── config.json              (model configuration)
    ├── generation_config.json   (generation settings)
    ├── model-00001-of-00004.safetensors  (sharded model weights)
    ├── model-00002-of-00004.safetensors
    ├── model-00003-of-00004.safetensors
    ├── model-00004-of-00004.safetensors
    ├── model.safetensors.index.json
    ├── special_tokens_map.json
    ├── tokenizer_config.json
    ├── tokenizer.json
    ├── tokenizer.model
    └── trainer_state.json

This directory IS your SFT checkpoint. Use this path as --model-path in the
RL training phase.

5. CALCULATING ACCURACY AFTER SFT
================================================================================

The codebase does NOT include a standalone SFT evaluation script. But you can
calculate SFT accuracy in TWO ways:

METHOD 1: USE THE RL CODEBASE IN EVAL-ONLY MODE
------------------------------------------------

The RL training code (main.py) evaluates the model at the start and
periodically during training. The FIRST evaluation (iteration 0) effectively
measures the SFT checkpoint's accuracy before any RL updates.

Look at the training logs for the first few iterations. The
"episode_success_rate" metric at the beginning (before any PPO updates are
applied) gives you the SFT accuracy.

When you run RL training, the first rollout collects 512 environment steps
using the model BEFORE any PPO update happens, so the reported success rate
in the initial iterations is your SFT baseline.

METHOD 2: WRITE A SIMPLE EVALUATION SCRIPT
-------------------------------------------

You can write a standalone evaluation script like this:

  -----------------------------------------------------------------------
  # eval_sft_numberline.py
  # Place this in VLM_PPO/ directory

  import torch
  import gymnasium as gym
  import gym_cards
  import numpy as np
  from PIL import Image
  from functools import partial

  from llava.constants import IMAGE_TOKEN_INDEX, DEFAULT_IMAGE_TOKEN
  from llava.conversation import conv_templates
  from llava.mm_utils import tokenizer_image_token
  from llava.model.language_model.llava_mistral import LlavaMistralForCausalLM
  from transformers import AutoTokenizer

  from a2c_ppo_acktr.rl_utils import get_prompt, text_projection

  # === CONFIGURATION ===
  MODEL_PATH = "/path/to/your/sft/checkpoint"  # <-- SET THIS
  NUM_EVAL_EPISODES = 200
  TEMPERATURE = 0.2
  MAX_NEW_TOKENS = 256
  ENV_NAME = "gym_cards/NumberLine-v0"

  # Load model
  tokenizer = AutoTokenizer.from_pretrained(MODEL_PATH)
  model = LlavaMistralForCausalLM.from_pretrained(MODEL_PATH)
  model.eval()
  model = model.cuda()

  vision_tower = model.get_vision_tower()
  if not vision_tower.is_loaded:
      vision_tower.load_model()
  vision_tower = vision_tower.cuda()
  image_processor = vision_tower.image_processor

  # Create environment
  env = gym.make(ENV_NAME)

  # Build prompt
  qs = get_prompt(ENV_NAME, action_only=False)
  qs = DEFAULT_IMAGE_TOKEN + "\n" + qs
  conv = conv_templates["llava_v1"].copy()
  conv.append_message(conv.roles[0], qs)
  conv.append_message(conv.roles[1], None)
  prompt = conv.get_prompt()

  successes = 0
  total_rewards = []

  for episode in range(NUM_EVAL_EPISODES):
      obs, info = env.reset()
      done = False
      truncated = False
      episode_reward = 0

      while not done and not truncated:
          # Process image
          pil_image = Image.fromarray(obs)
          image_tensor = image_processor.preprocess(
              pil_image, return_tensors='pt'
          )['pixel_values'].cuda().to(model.dtype)

          # Tokenize prompt
          input_ids = tokenizer_image_token(
              prompt, tokenizer, IMAGE_TOKEN_INDEX, return_tensors='pt'
          ).unsqueeze(0).cuda()

          # Prepare multimodal inputs
          _, _, _, _, inputs_embeds, _ = model.prepare_inputs_labels_for_multimodal(
              input_ids, None, None, None, None, image_tensor
          )

          # Generate
          with torch.inference_mode():
              outputs = model.generate(
                  inputs_embeds=inputs_embeds,
                  do_sample=True,
                  temperature=TEMPERATURE,
                  max_new_tokens=MAX_NEW_TOKENS,
                  use_cache=True,
                  pad_token_id=tokenizer.eos_token_id,
              )

          text_output = tokenizer.decode(outputs[0], skip_special_tokens=True)

          # Parse action
          action_idx = text_projection([text_output], ENV_NAME)
          action = action_idx.item()

          obs, reward, done, truncated, info = env.step(action)
          episode_reward += reward

          # Update prompt with new info (if needed for multi-step)
          qs = get_prompt(ENV_NAME, action_only=False, infos=[info])
          qs = DEFAULT_IMAGE_TOKEN + "\n" + qs
          conv = conv_templates["llava_v1"].copy()
          conv.append_message(conv.roles[0], qs)
          conv.append_message(conv.roles[1], None)
          prompt = conv.get_prompt()

      if episode_reward > 0:
          successes += 1
      total_rewards.append(episode_reward)

      if (episode + 1) % 50 == 0:
          print(f"Episode {episode+1}/{NUM_EVAL_EPISODES}, "
                f"Success Rate: {successes/(episode+1)*100:.1f}%, "
                f"Mean Reward: {np.mean(total_rewards):.3f}")

  print(f"\n=== FINAL SFT ACCURACY ===")
  print(f"Success Rate: {successes/NUM_EVAL_EPISODES*100:.1f}%")
  print(f"Mean Reward: {np.mean(total_rewards):.3f}")
  print(f"Episodes: {NUM_EVAL_EPISODES}")
  -----------------------------------------------------------------------

  Run with:
    cd VLM_PPO
    python eval_sft_numberline.py

EXPECTED SFT ACCURACY FOR NUMBERLINE:
  - Paper reports ~50.2% for LLaVA-SFT (with CoT)
  - Without CoT, accuracy drops significantly
  - This varies with SFT dataset size and quality

6. PHASE 2: RL TRAINING (PPO)
================================================================================

6A. HOW RL USES THE SFT CHECKPOINT
------------------------------------

1. The SFT checkpoint is loaded as the STARTING POINT for RL training
2. LoRA adapters are applied ON TOP of the SFT model (not full fine-tuning)
3. The SFT model provides the initial policy (pi_0) and is also used for
   the value function base
4. Key difference from SFT: the model now INTERACTS with the live environment
   and learns from REWARDS, not from static demonstrations

Loading flow in main.py:
  - If model_path contains "lora": loads base + LoRA weights, merges them
  - Otherwise: loads full model from model_path (your SFT checkpoint)
  - Then applies FRESH LoRA adapters for RL training
  - Also creates a 3-layer MLP value head on top of the model

6B. THE RL TRAINING LOOP STEP-BY-STEP
---------------------------------------

The training loop in main.py works as follows:

  Total env steps: 15000 (--num-env-steps)
  Steps per update: 512 (--num-steps)
  Num updates = 15000 / 512 / num_processes = ~15 updates (with 1 process)

  For each UPDATE iteration j:

    1. COLLECT DATA (on-policy rollout):
       For step = 0 to 511:
         a. Take current observation (image tensor)
         b. Process image through image_processor -> pixel_values
         c. Construct prompt: <image_token>\n<CoT prompt text>
         d. Tokenize prompt -> input_ids
         e. Call actor_critic.act():
            - llava_generate() generates text output with sampling
            - text_projection() parses output to find action ("-" or "+")
            - Returns: value estimate, output_ids, action, action_log_prob
         f. Execute action in environment -> obs, reward, done, info
         g. Store transition in rollouts buffer:
            (obs, output_ids, action, action_log_prob, value, reward, mask)
         h. If episode done:
            - Record episode reward and success (reward > 0 = success)
            - Environment auto-resets

    2. COMPUTE RETURNS:
       - Get V(s_final) from actor_critic.get_value()
       - Use GAE (Generalized Advantage Estimation) to compute advantages:
         delta_t = r_t + gamma * V(s_{t+1}) * mask - V(s_t)
         A_t = sum_{l=0}^{T-t} (gamma * lambda)^l * delta_l
       - Returns = Advantages + Values
       - gamma = 0.9, gae-lambda = 0.95

    3. PPO UPDATE:
       - Normalize advantages (subtract mean, divide by std)
       - For ppo_epoch = 4 epochs:
         - Sample mini-batches from rollout buffer
         - For each mini-batch:
           a. Re-evaluate actions: actor_critic.evaluate_actions()
              -> new values, new action_log_probs
           b. Compute ratio = exp(new_log_prob - old_log_prob)
           c. Compute clipped surrogate objective:
              surr1 = ratio * advantage
              surr2 = clip(ratio, 1-eps, 1+eps) * advantage
              action_loss = -min(surr1, surr2)
              * Safety: if any ratio > 10, use -surr2 only
           d. Compute value loss (clipped):
              v_clipped = v_old + clip(v_new - v_old, -eps, eps)
              value_loss = 0.5 * max(MSE(v_new, returns), MSE(v_clipped, returns))
           e. Total loss = value_loss_coef * value_loss + action_loss
           f. Backward pass with gradient accumulation (128 steps)
           g. Gradient clipping (max_grad_norm = 0.01)
           h. Optimizer step

    4. LEARNING RATE UPDATE:
       - Cosine annealing: 1e-5 -> 1e-9 over 25 steps

    5. LOGGING:
       - Print mean/median/min/max rewards, success rate
       - If wandb enabled: log all metrics

6C. THE PROMPT (WHAT THE VLM SEES)
------------------------------------

For NumberLine, the FULL prompt with CoT is constructed in rl_utils.py:

  "You are playing a game called number line. You will see a target number
   and a current number in the image. And your goal is to move the current
   number closer to the target by choosing either adding or subtracting one
   to the current number. Your response should be a valid json file in the
   following format:
   {
    "current number": "x",
    "target number": "x",
    "thoughts": "{first read out the current and target number, then think
                  carefully about which action to choose}",
    "action": "-" or "+"
   }"

This is prepended with the <image> token and wrapped in the LLaVA v1
conversation template:

  [INST] <image>
  <prompt text> [/INST]

The model then generates a JSON response like:
  {
    "current number": "1",
    "target number": "4",
    "thoughts": "The current number is 1 and the target is 4. Since 4 > 1,
                 I should add to move closer.",
    "action": "+"
  }

WITHOUT CoT (--action_only_prompt flag), the prompt becomes:
  "... Your response should be a valid json file in the following format:
   {
    "action": "-" or "+"
   }"

6D. HOW TEXT OUTPUT IS PARSED TO ACTIONS
-----------------------------------------

FILE: a2c_ppo_acktr/rl_utils.py -> text_projection()

For NumberLine:
  action_list = ["-", "+"]

The parsing logic:
  1. Find '"action":' in the output string
  2. Take everything after that
  3. Check which actions from action_list appear in that substring
  4. If exactly ONE action is found -> return its index (0 for "-", 1 for "+")
  5. If ZERO or MULTIPLE actions found -> return RANDOM index (exploration!)

This random fallback is important: if the model generates invalid output,
it becomes random exploration rather than crashing. Over training, the model
learns to generate valid outputs more consistently.

6E. REWARD FUNCTION
---------------------

NumberLine's reward is SHAPED (not just sparse):

  +1  : Current position == Target position (SUCCESS! Episode ends)
  -1  : Action moved current AWAY from or EQUAL distance to target
   0  : Action moved current CLOSER to target (but not there yet)

This shaping helps RL: the agent gets immediate negative feedback for bad
moves and learns faster than with a pure sparse reward (only +1 at end).

Additionally, if invalid text is generated (random fallback), the random
action has a 50% chance of being wrong -> -1 reward, pushing the model to
generate valid text.

6F. THE PPO ALGORITHM DETAILS
-------------------------------

FILE: a2c_ppo_acktr/algo/ppo.py

Standard clipped PPO with these specifics:

  HYPERPARAMETERS (for NumberLine):
    clip_param = 0.1 (epsilon for clipping)
    ppo_epoch = 4 (epochs per update)
    mini_batch_size = 1 (samples per mini-batch)
    grad_accum_steps = 128 (effective batch = 128)
    value_loss_coef = 0.5
    entropy_coef = 0.01 (note: not actually used in the loss!)
    max_grad_norm = 0.01
    gamma = 0.9 (discount factor)
    gae_lambda = 0.95

  LOSS COMPUTATION:
    advantages = returns - value_predictions
    advantages = (adv - mean(adv)) / (std(adv) + 1e-5)

    ratio = exp(log_prob_new - log_prob_old)
    surr1 = ratio * advantages
    surr2 = clip(ratio, 1-0.1, 1+0.1) * advantages

    # Safety check (from HuggingFace TRL):
    if any(ratio > 10):
        action_loss = -mean(surr2)   # use only clipped version
    else:
        action_loss = -mean(min(surr1, surr2))  # standard PPO

    value_loss = 0.5 * max(
        (values - returns)^2,
        (clip(values, v_old - eps, v_old + eps) - returns)^2
    )

    total_loss = 0.5 * value_loss + action_loss

6G. THE LAMBDA (THOUGHT-PROB-COEF) TRICK
------------------------------------------

FILE: a2c_ppo_acktr/llava_interface/interface.py -> llava_evaluate()

This is the KEY TECHNICAL CONTRIBUTION of the paper.

PROBLEM:
  When computing the policy gradient, we need log pi(action | state).
  But the VLM generates BOTH CoT tokens AND action tokens.
  The CoT portion has MUCH larger absolute log-probability than the
  action portion (e.g., -37.6 vs -0.4), drowning out the action signal.

SOLUTION:
  Split the log-probabilities at the "action": boundary:
    thought_log_prob = sum of log-probs for tokens BEFORE "action":
    action_tokens_log_prob = sum of log-probs for tokens AFTER "action":

  Apply scaling:
    total_log_prob = lambda * thought_log_prob + action_tokens_log_prob

  Where lambda = --thought-prob-coef = 0.5 for NumberLine

HOW THE BOUNDARY IS FOUND:
  - Hard-coded token IDs: [29908, 2467, 1115]
  - These encode the string '"action":' in the Mistral tokenizer
  - WARNING: These IDs are tokenizer-version-dependent!
  - If using a different tokenizer version, you MUST verify/change these IDs
  - To check: print(tokenizer.encode('"action":'))

WHAT HAPPENS IF "action": IS NOT FOUND:
  - Falls back to using the last 4 non-zero tokens as the "action" portion
  - If that also fails, returns fixed log-probs (-2 for total, -1 for action)
  - This prevents crashes from malformed outputs

6H. LORA CONFIGURATION
------------------------

FILE: main.py (lines around LoRA setup)

  LoraConfig(
      r=128,                    # LoRA rank (relatively large)
      lora_alpha=256,           # Scaling factor = alpha/r = 2
      target_modules=<all linear layers>,  # Found by find_all_linear_names()
      lora_dropout=0.05,        # Dropout for regularization
      bias="none",              # No bias training
      task_type="CAUSAL_LM",    # Causal language modeling
  )

WHAT LAYERS GET LoRA:
  - ALL linear layers in Mistral-7B (q_proj, k_proj, v_proj, o_proj,
    gate_proj, up_proj, down_proj)
  - ALL linear layers in the MLP projector
  - ALL linear layers in CLIP ViT vision encoder (when --train-vision all)
  - EXCLUDES: lm_head (output head), value_head, vision_resampler

WHY LoRA AND NOT FULL FINE-TUNING?
  - PPO requires storing old and new log-probs, which means effectively
    double the memory for full fine-tuning
  - LoRA dramatically reduces trainable parameters (~1-2% of total)
  - With r=128, still captures significant adaptation capacity
  - DeepSpeed ZeRO-2 + LoRA makes training feasible on consumer GPUs

6I. VALUE HEAD ARCHITECTURE
-----------------------------

FILE: a2c_ppo_acktr/model.py -> VLMValue class

  value_head = Sequential(
      Linear(4096, 1024),    # 4096 = Mistral hidden size
      ReLU(),
      Linear(1024, 512),
      ReLU(),
      Linear(512, 1)         # Scalar value output
  )

HOW IT WORKS:
  1. Take the LLaVA model's last hidden state at the LAST token position
  2. Pass it through the 3-layer MLP -> scalar V(s)
  3. This estimates "how good is the current state?" for GAE computation
  4. The value head is trained from scratch (randomly initialized)
  5. It's on the SAME device as the base model (dtype=float16)

  Method             Success Rate (%)    Notes
  ─────────────────────────────────────────────────────
  Random Policy       ~20%               2-action, need ~5 correct in a row

11. COMMON ISSUES AND TROUBLESHOOTING
================================================================================

ISSUE 1: "dejavu/DejaVuSans-Bold.ttf not found"
  FIX: Install the DejaVu font or download from the repo's dejavu.zip
       Place the font file so it's accessible from the working directory
       Or modify numberline.py to point to the correct font path

ISSUE 2: Tokenizer version mismatch (wrong action token IDs)
  SYMPTOMS: Training runs but success rate doesn't improve, or lambda
            scaling has no effect
  FIX: Verify token IDs (see Section 8, Tokenizer Warning)
       Update target tensor in interface.py

ISSUE 3: NCCL timeout with multiple GPUs
  FIX: Reduce num_processes in config_zero2.yaml to 1
       Or increase NCCL timeout: export NCCL_TIMEOUT=1800

ISSUE 4: Out of memory (OOM)
  FIX: Use --q4 flag for 4-bit quantization
       Reduce --max-new-tokens to 128
       Reduce --num-steps to 256
       Reduce batch size in finetune.sh

ISSUE 5: "xformers not found"
  FIX: pip install xformers (must be installed AFTER torch)
       Make sure torch version matches xformers compatibility

ISSUE 6: LLaVA import errors
  FIX: Make sure LLaVA is installed first: pip install -e LLaVA
       Check transformers version: should be 4.37.2

ISSUE 7: Accelerate/DeepSpeed config issues
  FIX: Run "accelerate config" to set up a fresh configuration
       Or ensure config_zero2.yaml matches your GPU setup

ISSUE 8: Model downloads fail
  FIX: Login to HuggingFace: huggingface-cli login
       Set HF_HOME for custom cache dir: export HF_HOME=/path/to/cache

ISSUE 9: SFT data not found
  FIX: Download from https://huggingface.co/LEVI-Project/sft-data
       Update --data_path and --image_folder in finetune.sh

ISSUE 10: Training loss is NaN
  FIX: This is handled in the code (exits with error message)
       Usually caused by learning rate too high or corrupted data
       Try reducing --init-lr to 5e-6
